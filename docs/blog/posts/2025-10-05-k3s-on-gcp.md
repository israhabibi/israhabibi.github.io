# K3s on GCP: Free-Tier Single Node to 3-Node Cluster

Running K3s (lightweight Kubernetes) on Google Cloud Platform — starting from a completely free `e2-micro` single-node setup and scaling to a 3-node cluster for around $15–$18 a month.

<!-- more -->

## Context

K3s strips Kubernetes down enough to run on very small VMs, which makes it a natural fit for GCP's free tier. This guide walks through both configurations and the networking bits that usually trip people up (firewall rules, kubeconfig rewrites, image registry auth).

---

## Cost Breakdown

| Setup | Monthly Cost | Notes |
|---|---|---|
| 1× `e2-micro` (US region) | **$0** | Always-free tier: 1 instance + 30 GB disk + 1 GB egress |
| 3-node cluster (1 server + 2 agents) | **~$15–18** | First node free; the other two are billed |
| Egress over 1 GB/month | Extra | Stop instances when idle to avoid surprises |

## Single-Node Setup (Free)

### 1. Create the VM

```bash
gcloud compute instances create k3s-node \
  --machine-type=e2-micro \
  --zone=us-central1-a \
  --image-family=ubuntu-2204-lts \
  --image-project=ubuntu-os-cloud \
  --boot-disk-size=30GB
```

Stick to a US region (`us-central1`, `us-east1`, `us-west1`) to stay inside the free tier.

### 2. Firewall Rules

```bash
# K3s API server
gcloud compute firewall-rules create allow-k3s-api \
  --allow tcp:6443 \
  --source-ranges=0.0.0.0/0

# HTTP / HTTPS ingress
gcloud compute firewall-rules create allow-http-https \
  --allow tcp:80,tcp:443 \
  --source-ranges=0.0.0.0/0
```

### 3. Install K3s

SSH in and run:

```bash
curl -sfL https://get.k3s.io | \
  INSTALL_K3S_EXEC="--write-kubeconfig-mode 644" sh -
```

`--write-kubeconfig-mode 644` makes `/etc/rancher/k3s/k3s.yaml` readable by your user — otherwise you'll be `sudo`'ing every kubectl call.

### 4. Access From Your Laptop

```bash
# On the VM
sudo cat /etc/rancher/k3s/k3s.yaml

# Copy locally, then replace the server address
# Original:  server: https://127.0.0.1:6443
# Replace:   server: https://<EXTERNAL_IP>:6443
```

Save it as `~/.kube/config-k3s-gcp` and test:

```bash
export KUBECONFIG=~/.kube/config-k3s-gcp
kubectl get nodes
```

### 5. Cost Optimization

Stop the instance when you're done — you keep the disk (free up to 30 GB) and pay nothing for compute while it's stopped.

```bash
gcloud compute instances stop k3s-node --zone=us-central1-a
gcloud compute instances start k3s-node --zone=us-central1-a
```

The external IP changes on restart unless you reserve a static one — budget that into the kubeconfig update step.

## 3-Node Cluster Setup

### 1. Create the Nodes

```bash
# Server
gcloud compute instances create k3s-server \
  --machine-type=e2-small \
  --zone=us-central1-a \
  --image-family=ubuntu-2204-lts \
  --image-project=ubuntu-os-cloud

# Agents
for i in 1 2; do
  gcloud compute instances create k3s-agent-$i \
    --machine-type=e2-small \
    --zone=us-central1-a \
    --image-family=ubuntu-2204-lts \
    --image-project=ubuntu-os-cloud
done
```

### 2. Additional Firewall Rules

K3s agents need to talk to the server on internal ports:

```bash
gcloud compute firewall-rules create allow-k3s-internal \
  --allow tcp:6443,tcp:8472,udp:8472,tcp:10250 \
  --source-tags=k3s \
  --target-tags=k3s
```

Tag all three VMs with `--tags=k3s` when you create them, or add the tag after with `gcloud compute instances add-tags`.

### 3. Install the Server

```bash
curl -sfL https://get.k3s.io | \
  INSTALL_K3S_EXEC="--write-kubeconfig-mode 644" sh -

# Get the join token
sudo cat /var/lib/rancher/k3s/server/node-token
```

### 4. Join the Agents

```bash
curl -sfL https://get.k3s.io | \
  K3S_URL=https://<SERVER_INTERNAL_IP>:6443 \
  K3S_TOKEN=<NODE_TOKEN> \
  sh -
```

Use the **internal** IP — agents-to-server traffic should stay inside the VPC.

## Flask App Deployment (End-to-End Example)

### Simple Flask App

```python
# app.py
from flask import Flask
app = Flask(__name__)

@app.route("/")
def home():
    return "Hello from K3s on GCP!"

@app.route("/health")
def health():
    return {"status": "ok"}, 200

if __name__ == "__main__":
    app.run(host="0.0.0.0", port=5000)
```

### Dockerfile

```dockerfile
FROM python:3.11-slim
WORKDIR /app
COPY app.py .
RUN pip install flask
EXPOSE 5000
CMD ["python", "app.py"]
```

### Push to GCR

```bash
docker build -t gcr.io/<PROJECT_ID>/flask-app:v1 .
gcloud auth configure-docker
docker push gcr.io/<PROJECT_ID>/flask-app:v1
```

### Deploy to K3s

```yaml
# flask-app.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: flask-app
spec:
  replicas: 2
  selector:
    matchLabels:
      app: flask-app
  template:
    metadata:
      labels:
        app: flask-app
    spec:
      containers:
        - name: flask-app
          image: gcr.io/<PROJECT_ID>/flask-app:v1
          ports:
            - containerPort: 5000
          livenessProbe:
            httpGet:
              path: /health
              port: 5000
---
apiVersion: v1
kind: Service
metadata:
  name: flask-app
spec:
  type: LoadBalancer
  selector:
    app: flask-app
  ports:
    - port: 80
      targetPort: 5000
```

```bash
kubectl apply -f flask-app.yaml
kubectl get svc flask-app
```

```bash

kubectl scale deployment flask-app --replicas=3

# Services
kubectl get svc
kubectl port-forward svc/flask-app 8080:80

# Namespaces
kubectl config set-context --current --namespace=<ns>
```

## Troubleshooting

!!! bug "Pods stuck in `ImagePullBackOff`"
    Agent nodes don't have GCR auth by default. Create a `docker-registry` secret or use Workload Identity / service account keys on each node.

!!! bug "Service `EXTERNAL-IP` stays `<pending>`"
    K3s ships with Traefik + Klipper load balancer, which uses the node's own IP. Make sure port 80/443 firewall rules are in place and hit the node's external IP directly.

!!! bug "Agent won't join"
    Check: (1) internal IP reachable between VMs, (2) `--source-tags=k3s` matches the instance tag, (3) token copied without trailing whitespace.

## Cleanup

### Single node
```bash
gcloud compute instances delete k3s-node --zone=us-central1-a
gcloud compute firewall-rules delete allow-k3s-api allow-http-https
```

### 3-node cluster
```bash
gcloud compute instances delete k3s-server k3s-agent-1 k3s-agent-2 --zone=us-central1-a
gcloud compute firewall-rules delete allow-k3s-api allow-http-https allow-k3s-internal
```

## Key Takeaways

- `e2-micro` in a US region = free forever; perfect for a learning cluster.
- Always replace `127.0.0.1` with the external IP when copying kubeconfig — that trips up almost everyone on day one.
- Stop instances when idle; compute is free while stopped, disk storage is also free under 30 GB.
- For 3-node setups, use **internal** IPs for agent-to-server join traffic, and tag VMs to scope firewall rules.
